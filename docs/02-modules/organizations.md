# Organizations Module

## Назначение

Модуль `organizations` отвечает за образовательные организации, группы, отделения, учебные периоды и членство пользователей.

---

## Основные задачи

- хранение и управление данными модуля;
- API для frontend;
- права доступа;
- связь с образовательной организацией;
- связь с личными кабинетами;
- уведомления при важных событиях;
- подготовка данных для аналитики.

---

## Основные модели

```text
Organization, OrganizationBranch, Department, StudyGroup, AcademicYear, AcademicPeriod, OrganizationMembership
```

---

## Основные сценарии

### Просмотр данных

Пользователь открывает соответствующий раздел личного кабинета и видит только те данные, к которым имеет доступ по роли и организации.

### Создание или изменение данных

Пользователь с нужной ролью создаёт или изменяет объект. Система проверяет права, валидирует данные, сохраняет изменения и при необходимости создаёт уведомления.

### Интеграция с другими модулями

Модуль должен быть связан с пользователями, организациями, уведомлениями и аналитикой.

---

## API

Endpoints должны находиться в пространстве:

```text
/api/v1/organizations/
```

Типовые действия:

```text
GET    /api/v1/organizations/
POST   /api/v1/organizations/
GET    /api/v1/organizations/{id}/
PATCH  /api/v1/organizations/{id}/
DELETE /api/v1/organizations/{id}/
```

---

## Права доступа

- учащийся видит только свои данные;
- родитель видит данные своих детей;
- преподаватель видит данные своих курсов и групп;
- директор видит данные своей организации;
- суперадмин видит данные всей платформы.

---

## Важные правила

- данные должны быть привязаны к организации;
- нельзя показывать данные одной организации пользователям другой;
- важные изменения должны попадать в историю или журнал действий;
- удаление значимых данных лучше делать через архивирование;
- API должен возвращать понятные ошибки доступа и валидации.
